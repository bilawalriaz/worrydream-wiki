# [[kurlander-1991-editable-graphical-histories-the-video|Kurlander]] 1992 - A History-Based Macro By Example System

## Core Thesis
The paper argues that for programming-by-example systems to be truly useful and accessible, they must provide a **persistent, editable, and comprehensible visual representation** of the demonstrated actions. The core problem is that most macro-by-example systems either lack a visual representation, provide only a textual one, or show only transient feedback for single steps. This lack makes the macro a "black box," hindering understanding, editing, and error diagnosis.

[[kurlander-1991-editable-graphical-histories-the-video|Kurlander]] and Feiner propose that the solution lies in an **"[[kurlander-1991-editable-graphical-histories-the-video|editable graphical]] history"** using a comic-strip metaphor. This representation doesn't just log actions; it renders them using the system's own graphical language, coalesces low-level commands into logical high-level operations, and provides focused context. This visual history becomes the primary interface for macro definition, selection, argument declaration, generalization, and editing. The nuance is that the system is *always recording* via an undo/redo mechanism, meaning macro creation is a reflective act of curation from an existing history, not a separate "record" mode.

## Historical Context
This work sits at the intersection of two streams in early 1990s HCI: **Programming by Example (PBE)** and **graphical [[perkins-1997-inventing-the-lisa-user-interface|user interface]] (GUI) extensibility**.

PBE systems like SmallStar (Halbert, 1984), Peridot ([[myers-1998-scripting-graphical-applications-by-demonstration|Myers]], 1988), and EAGER (Cypher, 1991) aimed to allow non-programmers to create macros by demonstration. However, they often struggled with the **representation problem**: how to show the inferred procedure in a way users could understand and modify. Some used text (Tinker, MIKE), some used single-step visual feedback (Peridot, Metamouse), and some required understanding code (Tweedle).

The state of the art in GUI extensibility, seen in systems like GNU Emacs, provided powerful macro capabilities but demanded programming skill. The gap was a system that combined the accessibility of demonstration with the transparency of a direct-manipulation visual representation. [[kurlander-1991-editable-graphical-histories-the-video|Kurlander]]'s work builds directly on his earlier research into editable [[kurlander-1991-editable-graphical-histories-the-video|graphical histories]] for general undo/redo, extending it specifically to solve the PBE representation problem with[[kurlander-1991-editable-graphical-histories-the-video|in the Chimera edi]]tor system.

## Key Contributions
1.  **Editable Graphical History as a Unified Interface:** The primary contribution is the design and implementation of a history mechanism that serves three functions simultaneously: undo/redo, a visual audit trail, and the workspace for macro creation. This collapses the distinction between using an application and programming it.
2.  **The Comic-Strip Metaphor for Action Representation:** The paper formalizes a technique for representing complex command sequences as a sequence of graphical panels. Key features include:
    *   **Logical Coalescing:** Grouping low-level physical actions (mouse clicks, key presses) into meaningful logical operations (e.g., "Add text label").
    *   **Focused Context:** Showing only the objects involved in an operation plus minimal context, rendering non-participating objects subtly to avoid visual clutter.
    *   **Interactivity:** Allowing panels to be selected, elaborated (expanded into sub-steps), and edited.
3.  **A Two-Phase, Non-Modal Macro Definition Process:** Separating demonstration from generalization. Phase one is just normal interaction; nothing special is recorded. Phase two involves *retrospectively* selecting steps from the persistent history to encapsulate, then augmenting them with arguments, generalizations, and control structures in a dedicated "Macro Builder" window. This aligns with how people often realize a task is repetitive only after performing it.
4.  **Visual Argument Declaration:** Users declare macro arguments by *graphically selecting* objects or properties *as they appear in the history panels* and naming them. This makes the argument's role in the procedure visually explicit. The system intelligently restricts selection to objects that exist at the macro's logical start point.
5.  **Generalization through [[hutchins-1985-direct-manipulation-interfaces|Direct Manipulation]]:** The paper describes facilities for generalizing a macro, such as converting a specific object reference to a variable, all performed by interacting with the graphical representation in the Macro Builder, not by writing code.

## Methodology
The methodology is **design research through system implementation**. The argument is structured as:
1.  **Problem Identification:[[kurlander-1991-editable-graphical-histories-the-video|** Critiquing exis]]ting PBE systems for their inadequate representations.
2.  **Proposed Solution:** Introducing the editable graphical history as a superior representational framework.
3.  **Implementation & Demonstration:** Detailing how this framework is extended to support all phases of macro lifecycle (definition, argumentation, generalization, editing) within the Chimera system.
4.  **Evaluation through Example:** Using a concrete, technical illustration workflow to walk the reader through the system's features, demonstrating its feasibility and intuitiveness.

The evidence is primarily **qualitative and exemplary**, relying on the reader's assessment of the provided figures and the described interaction sequences. It is a classic HCI systems paper, where the artifact itself—the implemented system—is the primary contribution and proof of concept.

## Influence
This paper is a foundational work in the lineage of **end-user programming** and **[[ingalls-1988-fabrik-a-visual-programming-environment|visual programming]] environments**. Its influence is seen in:

1.  **Later Macro and Scripting Systems:** The idea of a persistent, inspectable action history became common. Tools like **Adobe Photoshop's "Actions" palette** and **AppleScript's script editor** reflect the principle of recording and then editing a sequence of steps. The visual, panel-based representation directly foreshadows the step-by-step visual [[schachman-2012-alternative-programming-interfaces-for-alternative-programmers|programming interfaces]] in tools like **Scratch** (for logic) and, more directly, the **"History" panels** in modern creative software (e.g., Figma, Adobe XD) which, while often non-editable for macro creation, fulfill the audit trail function.
2.  **Programming by Demonstration (PBD) Research:** It significantly advanced PBD by tackling the representation head-on. It influenced subsequent work on making inferred procedures more inspectable and editable, a central challenge in PBD.
3.  **AI-Assisted Coding and Agent Systems:** The core idea—that complex actions can be monitored, represented, abstracted, and reused—is directly analogous to how modern AI coding assistants (like GitHub Copilot) or robotic process automation (RPA) tools observe user actions or code patterns to suggest or automate workflows. The "macro by example" is a primitive form of learning from demonstration.

## Modern Relevance
This paper's ideas are startlingly relevant to contemporary challenges in **AI, productivity, and tool design**:

1.  **The Foundation for "No-Code" and "Low-Code" Platforms:** The Macro Builder is a direct ancestor of modern visual workflow builders (e.g., Zapier, IFTTT, Microsoft Power Automate). These platforms allow users to chain actions together based on triggers, essentially creating macros across applications. [[kurlander-1991-editable-graphical-histories-the-video|Kurlander]]'s work anticipated the need for a [[cohn-2012-comics-linguistics-and-visual-language|visual language]] to represent these procedural chains.
2.  **AI and Large Language Models (LLMs) for Code Generation:** Modern AI assistants (GitHub Copilot, ChatGPT) can generate code fro[[kurlander-1991-editable-graphical-histories-the-video|m natural language]] or examples. [[kurlander-1991-editable-graphical-histories-the-video|Kurlander]]'s system can be seen as a **pre-LLM, deterministic approach** to the same problem: inferring a procedure from a de[[perkins-1997-inventing-the-lisa-user-interface|monstration. T]]he "editable graphical history" is a user interface for inspecting and correcting the AI's "inference" (the macro). The challenges of representing and allowing edits to an AI's inferred procedure are the same challenges [[kurlander-1991-editable-graphical-histories-the-video|Kurlander]] solved for a simpler, deterministic system.
3.  **Knowledge Management and Replayable Workflows:** In research, data science, and content creation, reproducibility is key. The editable history is a **provenance-tracking mechanism**. It creates a reproducible record of how a result was achieved. The macro is a named, reusable workflow. This directly relates to modern "computational notebook" environments (Jupyter) and version control for creative work.
4.  **The "Agent" Paradigm:** An AI agent that observes user behavior to automate tasks is performing *inference on a stream of actions*. This paper provides a seminal design for **how such an inferred sequence should be represented to the user for validation, editing, and trust**. The comic-strip metaphor offers a compelling alternative to a simple list of commands or a block of generated code for visualizing an agent's plan.

## Key Quotes
1.  > "Without a visual representation, it is impossible to review the operations that compose the macro. When there is an error in such a macro, the macro must be demonstrated once again from scratch."
    *   **Analysis:** This states the core usability problem starkly. It identifies the lack of inspectability and the high cost of error correction as primary failures of existing systems, framing the work as a solution to a fundamental friction in human-computer interaction.

2.  > "In our system, operations are always being recorded by an undo/redo mechanism. When users realize that a set of operations that they had performed are generally useful, they can always open up a history window and encapsulate the interesting operations into a macro."
    *   **Analysis:** This describes the key philosophical shift from "record mode" to "always-on reflection." It aligns tool capability with natural human cognition—we often recognize patterns retrospectively. This design choice eliminates modal friction and integrates programming seamlessly into the flow of work.

3.  > "Related commands are coalesced into single panels by pattern matching rules. This makes the histories more compact, but also makes them easier to interpret, since instead of showing physical commands the system shows logical commands."
    *   **Analysis:** This reveals the intelligence behind the representation. The system isn't just a naive logger; it performs abstraction. This transforms a low-level event stream into a human-centric narrative of intent, which is essential for comprehension and reuse.

4.  > "To edit the demonstrated procedure, the user can either textually edit these steps or the resulting LISP procedure." (from the discussion of Tinker)
    *   **Analysis:** This quote, describing a *competing* system, serves to highlight [[kurlander-1991-editable-graphical-histories-the-video|Kurlander]]'s contribution. It implicitly argues that textual editing—whether of step logs or code—is a barrier that his graphical, direct-manipulation approach aims to remove.

5.  > "Argument declarations are placed at the beginning of our macros, just as they appear at the beginning of traditional procedures."
    *   **Analysis:** This shows the system's careful bridging of paradigms. While the interaction is novel and visual, it deliberately maps onto the established structure of procedural programming (declarations first), aiding transfer of knowledge for users with some technical background and grounding the innovation in familiar concepts.
